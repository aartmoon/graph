# PageRank для графа, который не помещается в оперативную память

В этой работе я реализовал PageRank для большого ориентированного невзвешенного графа. Основное ограничение задачи — граф нельзя целиком загрузить в память, поэтому рёбра, степени вершин и два массива рангов хранятся на диске. В оперативную память попадает только обрабатываемая часть данных.

Решение рассчитано на один компьютер с SSD и небольшим Java heap, например `-Xmx128m`. Для использования нескольких ядер основные фазы каждой итерации PageRank выполняются параллельно.

## Коротко о результате

- язык: Java 21;
- вход и выход: CSV;
- исходные ID вершин: любые значения `int32`, в том числе отрицательные и разреженные;
- внутреннее представление: плотные ID от `0` до `V - 1` и бинарные файлы на диске;
- алгоритм обработки: схема scatter/gather, близкая по идее к X-Stream;
- память: зависит от размера блока и числа одновременно работающих потоков, а не от полного числа рёбер;
- гипер-узлы не разворачиваются в большой список смежности;
- дублирующиеся рёбра удаляются, так как граф считается невзвешенным и простым;
- результат воспроизводим при одинаковых входных данных и параметрах запуска.

## Быстрый запуск

Нужен JDK 21. На Ubuntu 22.04 его можно установить так:

```bash
sudo apt update
sudo apt install openjdk-21-jdk
```

Сборка и тесты:

```bash
./gradlew clean test jar
```

Запуск на примере из папки `data`:

```bash
java -Xmx128m -jar build/libs/largegraph-pagerank-0.1.0.jar \
  --input data/edges.csv \
  --output output/pagerank.csv \
  --workdir work \

После завершения результат будет находиться в `output/pagerank.csv`. Входной файл, выходной файл и рабочая директория должны быть разными путями. Выходной файл нельзя размещать внутри `workdir`.

  --threads 8
```

## Почему я выбрал PageRank

PageRank оценивает важность вершины не только по числу входящих рёбер, но и по важности вершин, которые на неё ссылаются. Это подходит, например, для ранжирования веб-страниц, пользователей социальной сети, научных публикаций или объектов в графе рекомендаций.

Я выбрал эту метрику по трём причинам:

1. Она применяется сразу ко всему графу, как и требуется в задании.
2. Один шаг алгоритма можно посчитать последовательным проходом по рёбрам. Для этого не нужен случайный доступ ко всему списку смежности.
3. PageRank является итерационным алгоритмом, поэтому его удобно разделить на независимые задачи и выполнять в несколько потоков.

Для вершины `v` очередное значение считается по формуле:

```text
next[v] = (1 - d) / V
        + d * danglingMass / V
        + d * sum(rank[u] / outDegree[u])
```

Здесь:

- `d` — коэффициент затухания, по умолчанию `0.85`;
- `V` — число вершин;
- сумма берётся по рёбрам `u -> v`;
- `danglingMass` — сумма рангов вершин без исходящих рёбер.

Вершины без исходящих рёбер нельзя просто пропустить, иначе на каждой итерации терялась бы часть общей массы. Поэтому их ранг равномерно распределяется между всеми вершинами.

Начальное значение каждой вершины равно `1 / V`. Успешное вычисление заканчивается, когда L1-разность соседних векторов становится меньше `epsilon`. Если этого не произошло за `max-iterations`, программа сообщает об ошибке и не выдаёт недосчитанный результат.

## Рассмотренные варианты хранения графа

Я сравнил три варианта.

| Вариант | Плюсы | Почему выбран или отклонён |
| --- | --- | --- |
| CSR или список смежности в памяти | Мало накладных расходов и быстрый проход по рёбрам | Не подходит под главное условие: рёбра, смещения, степени и ранги должны поместиться в RAM |
| Подход GraphChi со скользящими шардами | Хорошая локальность дисковых обращений, известный подход для обработки больших графов на одном узле | Требует более сложной подготовки шардов и логики обновлений |
| Scatter/gather с рёбрами, разбитыми по исходной вершине | Можно читать данные последовательно, ограничивать память размером блока и естественно распараллелить вычисления | Этот вариант реализован в проекте |

Выбранный способ не самый экономный по дисковому вводу-выводу: на каждой итерации создаются сообщения для рёбер. Зато он относительно простой, не требует хранить граф в heap и нормально обрабатывает вершины с очень большой степенью.

## Как устроено решение

Работа делится на подготовку графа и итерации PageRank.

### 1. Подготовка графа

Исходные ID нельзя использовать как индексы массива. Например, для единственного ребра `100 -> 1000000` реально существуют две наблюдаемые вершины, а не миллион. При этом хранить `HashMap<originalId, denseId>` для всех вершин тоже нельзя: на большом графе он сам может занять всю память.

Поэтому перенумерация выполняется через внешнюю сортировку:

1. CSV читается построчно. Для каждого конца ребра записывается тройка `(originalId, edgeId, side)`.
2. Эти записи сортируются на диске по исходному ID вершины.
3. В одном последовательном проходе уникальным вершинам назначаются плотные ID `0..V-1`. Одновременно создаётся `vertices.bin`, который сохраняет соответствие плотного и исходного ID.
4. Назначения сортируются по `(edgeId, side)`, после чего восстанавливаются рёбра уже с плотными ID.
5. Плотные рёбра сортируются по `(from, to)`. На слиянии одинаковые рёбра удаляются.
6. Отсортированные рёбра распределяются по исходным блокам, одновременно считается исходящая степень каждой вершины.
7. Файл текущих рангов заполняется значением `1 / V`.

Внешняя сортировка сделана обычным способом: ограниченная порция записей сортируется в памяти и записывается как серия, затем серии сливаются. Числа хранятся в примитивных массивах, чтобы не создавать миллионы Java-объектов.

### 2. Одна итерация PageRank

Итерация состоит из трёх фаз.

**Подсчёт массы висячих вершин.** Файлы рангов и степеней читаются блоками. Диапазоны блоков распределяются между потоками, после чего частичные суммы складываются.

**Scatter.** Для каждого ребра `u -> v` вычисляется вклад

```text
d * rank[u] / outDegree[u]
```

и на диск записывается сообщение `(v, contribution)`. Сообщения группируются по диапазонам целевых вершин. Число диапазонов ограничено 8192, поэтому программа не пытается открыть отдельный файл для каждой вершины или каждой маленькой партиции.

**Gather.** Поток загружает один диапазон целевых вершин, заполняет его базовым значением, добавляет все сообщения для этого диапазона и записывает новые ранги. В этот же проход считаются сумма рангов и L1-разность с предыдущей итерацией.

После итерации файлы текущих и новых рангов меняются ролями. Временные сообщения удаляются даже при ошибке.

### Рабочие файлы

Во время выполнения рабочая директория выглядит примерно так:

```text
work/
  vertices.bin
  vertex/
    out_degree.bin
    rank_current.bin
    rank_next.bin
  edges_by_source/
    src-part-00000.bin
    src-part-00001.bin
  messages/
    iter-000001/
      worker-00000/
        msg-bucket-00000.bin
```

Основные бинарные записи имеют фиксированный размер:

```text
vertices.bin:        int originalId
edge:                int denseFrom, int denseTo       # 8 байт
message:             int denseTo, double contribution # 12 байт
rank:                double                            # 8 байт
out-degree:          int                               # 4 байта
```

После успешной записи итогового CSV управляемые файлы из `workdir` удаляются. Повторное использование подготовленного графа и продолжение с контрольной точки пока не реализованы.

## Многопоточность и гипер-узлы

Фазы подсчёта висячей массы, scatter и gather используют фиксированный пул потоков. Количество реально запущенных потоков не превышает:

- значение `--threads`;
- число доступных процессоров;
- безопасный предел по heap;
- предел по количеству временных файлов.

Если запрошенная конфигурация не помещается в память, программа уменьшает параллелизм и пишет это в лог. Если не помещается даже одна задача, запуск завершается понятной ошибкой.

Тяжёлые части обрабатываются раньше. Большая исходная партиция режется на срезы размером `--scatter-slice-mb`, после чего срезы жадно распределяются между рабочими потоками. Поэтому гипер-узел не превращается в огромный массив соседей и его рёбра могут обрабатываться несколькими потоками.

Ускорение не обязано быть линейным: потоки конкурируют за SSD, а маленький граф или один небольшой срез просто не дадут достаточно параллельной работы.

## Оценка сложности

Обозначения:

- `V` — число уникальных вершин, встретившихся в рёбрах;
- `E_in` — число строк-рёбер во входном CSV;
- `E` — число рёбер после удаления дубликатов;
- `C` — `chunk-size`;
- `T` — число одновременно работающих задач;
- `K` — число итераций до сходимости.

| Этап | Время | Последовательный дисковый ввод-вывод |
| --- | --- | --- |
| Чтение CSV | `O(E_in)` | `O(E_in)` |
| Внешняя сортировка и перенумерация | `O(E_in log E_in)` сравнений | несколько проходов объёма `O(E_in)` |
| Одна итерация PageRank | `O(V + E)` | `O(V + E)` |
| Все итерации | `O(K(V + E))` | `O(K(V + E))` |

Эти оценки одинаковы по порядку для среднего и худшего случая. На практике время сильнее всего зависит от числа проходов внешней сортировки, количества итераций и скорости SSD.

Память во время PageRank можно оценить как

```text
O(C * T + размер диапазона gather + служебные метаданные).
```

Полные массивы `rank[V]`, `nextRank[V]`, `outDegree[V]`, список всех рёбер и глобальная таблица перенумерации в heap не создаются. Внешняя сортировка имеет отдельный ограниченный буфер — не более 250 000 записей в одной серии.

Постоянные данные на диске занимают `O(V + E)`. На одной итерации сообщения требуют ещё примерно `12 * E` байт без учёта блоков файловой системы. Подготовка временно хранит несколько представлений рёбер, поэтому перед каждым крупным этапом программа проверяет свободное место.

Для heap 128 МБ разумно начать с `--chunk-size` от `10000` до `100000`. Слишком большой блок не ускорит программу, если из-за него придётся уменьшить число потоков.

## Корректность и точность

При `0 < d < 1` телепортационная часть PageRank делает стационарное распределение единственным. Оператор одной итерации сжимает L1-расстояние не хуже чем в `d` раз, поэтому метод сходится геометрически даже для графа с циклами и висячими вершинами.

Критерий остановки использует разность двух соседних приближений. Если `delta < epsilon`, то без учёта погрешности `double` можно оценить оставшуюся ошибку как

```text
||rank - rank*||₁ <= d / (1 - d) * epsilon.
```

При стандартных `d = 0.85` и `epsilon = 1e-8` эта оценка равна примерно `5.67e-8`. Это верхняя оценка, фактическая ошибка обычно меньше.

Кроме условия сходимости, реализация проверяет:

- сумма рангов после каждой итерации отличается от 1 не больше чем на `1e-6`;
- все ранги конечные и неотрицательные;
- scatter создал ровно по одному сообщению на каждое сохранённое ребро;
- gather прочитал столько же сообщений, сколько записал scatter;
- размеры бинарных файлов кратны размеру записи;
- все ID и диапазоны находятся в допустимых границах.

Тесты покрывают цикл с одинаковыми рангами, висячую вершину, разреженные и отрицательные ID, дубликаты рёбер, границы блоков, гипер-узел, повторяемость результата и сравнение с простой реализацией PageRank в памяти на небольшом графе.

## Формат данных

Используются первые две колонки CSV:

```csv
from,to
1,2
2,3
3,1
4,1
```

Дополнительные колонки разрешены и игнорируются:

```csv
from,to,comment
1,2,first edge
2,3,second edge
```

Допускаются заголовок `from,to`, пустые строки, пробелы вокруг ID и UTF-8 BOM в начале файла. ID должны помещаться в `int32`. Полный синтаксис CSV с экранированными запятыми и кавычками не поддерживается, потому что парсер читает только два первых числовых поля.

Результат имеет вид:

```csv
vertex,rank
1,0.33260446915035513
2,0.320213802437462
3,0.30968172841218256
4,0.037500000000000006
```

Строки идут по возрастанию исходного ID вершины. Итог сначала полностью записывается во временный соседний файл, а затем заменяет целевой файл атомарным перемещением, если файловая система это поддерживает.

## Ограничения

- Вершины задаются только рёбрами. Изолированную вершину без единого ребра этот формат входа представить не может.
- Число наблюдаемых вершин должно помещаться в диапазон плотного `int` ID, а исходящая степень одной вершины — в `int32`.
- На каждой итерации записывается `O(E)` сообщений. Для графа с большим числом рёбер нужен запас места на SSD.
- Подготовка делает несколько полных проходов по временным файлам, поэтому на маленьком графе обычный вариант в памяти будет заметно быстрее.
- Нет сжатия, контрольных точек и продолжения прерванного вычисления.
- Рабочая директория не переиспользуется между запусками и очищается после успеха.
- Точность ограничена арифметикой `double`; побитовая идентичность гарантируется только для одинакового входа и одинаковой конфигурации.
- Программа рассчитана на один вычислительный узел и не распределяет работу между несколькими машинами.

Главный компромисс решения — дополнительный дисковый ввод-вывод в обмен на ограниченное потребление оперативной памяти. Для условия задания это оправдано: программа может обработать граф больше heap, использует несколько ядер и не ломается на разреженных ID или гипер-узлах.
